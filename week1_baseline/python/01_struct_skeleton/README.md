# The Struct Skeleton (Python port)

Python port of `week1_baseline/ruby/01_struct_skeleton`. Same behavior, same
shared `.boukensha/` config directory — see the Ruby README for full design
rationale. This file documents the Python specifics and the differences
from Ruby worth knowing about.

Let's define data structures for specific data we need to constantly pass around.
- `boukensha.Tool`
- `boukensha.Message`
- `boukensha.Context`

Ruby represents these as `Struct`s — lightweight, mutable, readable for
learning (in practice you'd probably use full classes). Python's closest
equivalent is `@dataclass`: also lightweight and mutable (fields stay
settable, same as a Ruby Struct), with type hints and a custom `__str__`.

`config.py` in this step is the same `Config` class as `00_config`'s,
**minus** the `PROMPTS_DIR` constant — this step ships no `prompts/`
directory of its own, so `Player.system_prompt(...)` is only ever called
with `user_prompts_dir=`, relying entirely on the shared
`.boukensha/prompts/player/system.md` override.

## Code Changes

| File | Purpose |
|------|---------|
| `lib/boukensha/config.py` | `Config` class (no `PROMPTS_DIR` this step) |
| `lib/boukensha/tasks/base.py` | abstract `Base` task — verbatim copy from `00_config` |
| `lib/boukensha/tasks/player.py` | concrete `Player(Base)` — verbatim copy from `00_config` |
| `lib/boukensha/tool.py` | `@dataclass Tool` |
| `lib/boukensha/message.py` | `@dataclass Message` |
| `lib/boukensha/context.py` | `Context` class |
| `lib/boukensha/__init__.py` | top-level package exports |
| `examples/example.py` | runnable smoke-test |

## Data Structures

### `boukensha.Tool`
| Field | Description |
|---|---|
| `name` | The name of the tool |
| `description` | Shown to the agent so it knows when to invoke the tool |
| `parameters` | The arguments that need to be passed in |
| `block` | The actual code (a `Callable`) that runs when the tool is called |

### `boukensha.Message`
A single unit of conversation between the user and the agent.

| Field | Description |
|---|---|
| `role` | Who is speaking — user, assistant, or tool_result |
| `content` | The text generated by the agent or provided by the user |
| `tool_use_id` | Links a tool result back to the specific tool call that requested it. The API requires this pairing to be exact. |

### `boukensha.Context`
Holds everything BOUKENSHA needs to make an API call. Nothing lives outside of this.

| Field | Description |
|---|---|
| `task` | The task class driving this run (e.g. `Player`) |
| `system` | Tells the agent what its job is. Sent as a separate field in the API call, never inside the messages array. |
| `messages` | The full conversation history, replayed on every turn |
| `tools` | The registered tools the agent is allowed to invoke |

Note: the Ruby README for this step also documents a `token_budget`/`budget=`
field on `Context` — that's aspirational documentation for a future step;
the actual `context.rb` (and this Python port) has no such attribute yet.
This README only documents what `Context.__str__` produces today.

## Differences from Ruby (cosmetic, not behavioral)

- **String-only keys**, inherited from the `00_config` port: `ctx.add_message("user", ...)` not `:user`, tool `parameters` use string keys (`{"direction": {...}}` not `{direction: {...}}`).
- **List repr**: `Tool.__str__` prints `params=['direction']` (Python list-of-strings repr) where Ruby prints `params=[:direction]` (symbol array) — purely a text-formatting difference, not a functional one.
- **`Context.tool_count`/`turn_count`** are Python `@property` (read as `ctx.tool_count`, no parens), matching Ruby's endless-method call style.

## Run Example

```bash
./week1_baseline/bin/python/01_struct_skeleton
```

Expected output (values from your `.boukensha/`):

```
=== Boukensha Step 1: Struct Skeleton ===

Config:   #<Boukensha::Config dir=/home/andrew/Sites/Claude-Code-Camp/.boukensha tasks=player>
Context:  #<Context task=player turns=2 tools=1>
Tool:     #<Tool name=move description=Move the player in a direction (north, so params=['direction']>
Messages:
  #<Message role=user content=Explore north and tell me what you find....>
  #<Message role=assistant content=Sure, let me head north and take a look....>
```
